# LemonPay 유비쿼터스 언어 (Ubiquitous Language)

> 멀티통화 간편결제 시스템 - 팀 공용 용어 사전
> 버전: 1.1 | 최종 수정일: 2026-03-08 | 상태: 확정

---

## 1. 문서 목적

DDD(Domain-Driven Design)에서 유비쿼터스 언어란, 팀 전체(백엔드, 프론트엔드, 기획)가 동일한 개념을 동일한 단어로 표현하기 위한 공용 어휘이다. 이 문서는 LemonPay 프로젝트의 모든 구성원이 코드, 문서, 커뮤니케이션에서 일관된 용어를 사용하도록 정의한다.

**원칙**:
- 백엔드 코드(클래스명, 필드명)와 이 문서의 영문 용어가 일치해야 한다.
- 프론트엔드 UI 표시 문구는 "사용자 표시" 컬럼을 따른다.
- 팀 내 커뮤니케이션에서는 한국어 용어를 사용한다.
- 혼용 금지 용어(6장)에 명시된 용어는 반드시 지정된 표현만 사용한다.

## 2. 도메인별 핵심 용어

### 2.1 회원 (Member) 도메인

| 영문 용어 | 한국어 용어 | 정의 | BE 코드 | 사용자 표시 |
|----------|-----------|------|---------|-----------|
| Member | 회원 | LemonPay 서비스에 가입한 사용자. 가입 시 지갑이 자동 생성된다. | `Member` 엔티티 | - |
| Email | 이메일 | 회원의 로그인 식별자. 시스템 내 유일해야 한다. | `Member.email` | "이메일" |
| Phone | 휴대폰 번호 | E.164 국제 표준 형식으로 저장 (예: `+821012345678`). 표현 계층에서 `010-1234-5678`로 포맷한다. | `Member.phone` | "010-1234-5678" |
| Registration | 회원가입 | 신규 사용자가 계정을 생성하는 행위. 회원 생성과 동시에 지갑이 자동 생성된다 (FR-001). | `MemberService.register()` | "회원가입" |
| Authentication | 인증 | 회원이 본인임을 확인하는 절차. 세션 또는 토큰 기반. | `AuthService` | "로그인" |

### 2.2 지갑 (Wallet) 도메인

| 영문 용어 | 한국어 용어 | 정의 | BE 코드 | 사용자 표시 |
|----------|-----------|------|---------|-----------|
| Wallet | 지갑 | 회원이 보유한 멀티통화 자금 보관소. 회원당 1개 (현재 1:1, 향후 1:N 확장 고려). | `Wallet` 엔티티 | "내 지갑" |
| Primary Wallet | 주 지갑 | 회원의 기본 지갑. 1:N 전환 시 기본 결제/충전 대상이 되는 지갑. | `Wallet.isPrimary` | "기본 지갑" |
| Product Code | 상품 코드 | 지갑의 상품 유형과 버전을 식별하는 코드. ISO 20022 스타일 `{TYPE4}-V{VERSION}` 형식을 따른다. 예: `BASC-V1`. 향후 `wallet_product` 테이블로 분리 예정. | `Wallet.productCode` | "기본 지갑" / "여행 지갑" / "프리미엄 지갑" |
| Balance | 잔액 | 특정 통화의 사용 가능 금액. 스냅샷 잔액과 원장 합산이 항상 일치해야 한다. | `WalletBalance.balance` | "₩50,000" / "$100.00" |
| Wallet Balance | 통화별 잔액 | 지갑 내 특정 통화의 잔액 레코드. 지갑과 통화의 조합으로 유일하다. | `WalletBalance` 엔티티 | 통화별 금액 표시 |
| Charge | 충전 | 외부에서 지갑으로 KRW를 넣는 행위. 원장에 CHARGE/CREDIT으로 기록된다. | `WalletService.charge()` | "충전" |
| Snapshot Balance | 스냅샷 잔액 | 빠른 조회를 위해 `wallet_balance`에 미리 계산하여 저장한 현재 잔액. 원장 INSERT와 원자적으로 갱신된다. | `WalletBalance.balance` | - (내부 용어) |

#### 상품 코드 (Product Code) 체계

상품 코드는 ISO 20022 스타일의 `{TYPE4}-V{VERSION}` 형식을 사용한다.

**TYPE 코드표**:

| TYPE 코드 | 상품명 | 사용자 표시 | 설명 |
|----------|-------|-----------|------|
| `BASC` | 기본 지갑 | "기본 지갑" | 회원가입 시 기본 제공되는 상품 |
| `TRVL` | 여행 지갑 | "여행 지갑" | 여행 특화 상품 (환전 수수료 우대 등) |
| `PREM` | 프리미엄 지갑 | "프리미엄 지갑" | 프리미엄 회원 전용 상품 |

**VERSION 규칙**:
- 약관 또는 혜택이 변경될 때 VERSION을 증가시킨다 (예: `BASC-V1` -> `BASC-V2`)
- 기존 가입자는 가입 시점의 버전을 유지한다 (소급 적용 없음)
- 신규 가입자는 해당 상품의 최신 버전이 적용된다
- VERSION은 정수로만 증가한다 (V1, V2, V3, ...)

### 2.3 원장 (Ledger) 도메인

| 영문 용어 | 한국어 용어 | 정의 | BE 코드 | 사용자 표시 |
|----------|-----------|------|---------|-----------|
| Ledger | 원장 | 모든 잔액 변동의 불변 기록 저장소. INSERT만 허용하며 UPDATE/DELETE는 금지된다. | `ledger_entry` 테이블 | - (내부 용어) |
| Ledger Entry | 원장 항목 | 단일 잔액 변동의 불변 기록. 모든 입금 또는 출금은 정확히 하나의 항목을 생성한다. | `LedgerEntry` 엔티티 | - (내부 용어) |
| Direction | 방향 | 자금 이동의 방향. CREDIT(입금) 또는 DEBIT(출금). | `LedgerEntry.direction` | "+" / "-" |
| Credit | 입금 | 잔액이 증가하는 방향. 충전, 환전 입금, 결제 취소 환불 등. | `Direction.CREDIT` | "+" 접두사 (녹색) |
| Debit | 출금 | 잔액이 감소하는 방향. 결제, 환전 출금, 출금 등. | `Direction.DEBIT` | "-" 접두사 (빨간색) |
| Balance After | 거래 후 잔액 | 해당 원장 항목 처리 후의 잔액. 조회 성능과 검증용으로 기록한다. | `LedgerEntry.balanceAfter` | - (내부 용어) |
| Transaction ID | 거래 ID | 하나의 비즈니스 거래를 식별하는 고유 ID. 환전의 경우 EXCHANGE_OUT과 EXCHANGE_IN이 동일한 거래 ID를 공유한다. | `LedgerEntry.transactionId` | - (내부 용어) |
| Reconciliation | 대사 | 스냅샷 잔액과 원장 항목 합산 간의 정합성을 검증하는 정기 프로세스. 불일치 감지 시 로그를 기록하며 자동 보정은 하지 않는다. | 스케줄러 | - (내부 용어) |

### 2.4 결제 (Payment) 도메인

| 영문 용어 | 한국어 용어 | 정의 | BE 코드 | 사용자 표시 |
|----------|-----------|------|---------|-----------|
| Payment | 결제 | 회원이 가맹점에 대금을 지불하는 행위. 다통화 결제 시 Saga 패턴으로 처리된다. | `PaymentTransaction` 엔티티 | "결제" |
| Payment Transaction | 결제 거래 | 하나의 결제 요청에 대한 비즈니스 정보 전체. 금액, 통화, 상태, 가맹점 정보를 포함한다. | `PaymentTransaction` | 거래 내역 1건 |
| Merchant | 가맹점 | 결제를 수취하는 사업자. 코드에서는 `merchant`를 사용하되, UI에서는 상호명만 직접 표시한다. | `PaymentTransaction.merchantId` | 상호명 표시 (예: "Amazon") |
| Settlement | 정산 | 다통화 결제에서 실제 정산된 금액과 통화 정보. 적용된 환율을 포함한다. | `PaymentTransaction.settlementAmount` | "예상 차감: 13,500원" |
| Idempotency Key | 멱등성 키 | 동일 결제 요청의 중복 처리를 방지하기 위한 고유 식별자. 24시간 유효하다. | `PaymentTransaction.idempotencyKey` | - (내부 용어) |
| Compensation | 보상 | 실패한 Saga 단계의 효과를 되돌리는 역연산. 예: 차감된 잔액 복원. 보상 항목은 새로운 원장 항목으로 기록된다. | Saga 오케스트레이터 | - (내부 용어) |

### 2.5 환전 (Exchange) 도메인

| 영문 용어 | 한국어 용어 | 정의 | BE 코드 | 사용자 표시 |
|----------|-----------|------|---------|-----------|
| Exchange | 환전 | 한 통화에서 다른 통화로 자금을 변환하는 행위. 원장에 EXCHANGE_OUT/EXCHANGE_IN 쌍으로 기록된다. | `ExchangeTransaction` 엔티티 | "환전" |
| Exchange Rate | 환율 | 두 통화 간의 변환 비율. BigDecimal로 관리하며 부동소수점은 사용하지 않는다. | `ExchangeTransaction.exchangeRate` | "KRW 1 = USD 0.0007" |
| Rate Source | 환율 출처 | 환율을 획득한 방식. REALTIME(실시간 API), CACHED(TTL 이내 캐시), FALLBACK(만료 캐시/DB). | `ExchangeTransaction.rateSource` | FALLBACK 시 경고 배너 표시 |
| Source Currency | 출금 통화 | 환전에서 차감되는 통화. | `ExchangeTransaction.sourceCurrency` | "-₩10,000" |
| Target Currency | 입금 통화 | 환전에서 증가하는 통화. | `ExchangeTransaction.targetCurrency` | "+$7.40" |
| Circuit Breaker | 회로 차단기 | 비정상 외부 의존성(환율 API)에 대한 호출을 단락시켜 장애 전파를 방지하는 안정성 패턴. | Resilience4j | 장애 시 경고 배너 |
| Fallback | 폴백 | Circuit Breaker OPEN 상태에서 만료된 캐시 환율 또는 DB 마지막 저장 환율을 반환하는 대체 로직. | fallback 메서드 | "최근 환율(XX:XX 기준) 사용" |

## 3. 상태값 정의

### 3.1 회원 상태 (MemberStatus)

| 코드값 | 한국어 | 사용자 표시 | 정의 | 전이 규칙 |
|-------|-------|-----------|------|----------|
| `ACTIVE` | 활성 | 별도 표시 없음 (기본 상태) | 정상적으로 서비스를 이용할 수 있는 상태 | 초기 상태. SUSPENDED, CLOSED로 전이 가능 |
| `SUSPENDED` | 이용 정지 | "이용 정지" | 정책 위반 또는 관리자 조치로 서비스 이용이 제한된 상태. 조건 충족 시 ACTIVE로 복귀 가능 | ACTIVE에서 전이. ACTIVE 또는 CLOSED로 전이 가능 |
| `CLOSED` | 해지 | "탈퇴" | 회원이 탈퇴하거나 영구 해지된 상태. 논리 삭제 (물리 삭제 없음) | 최종 상태. 다른 상태로 전이 불가 |

```
ACTIVE ---[정책 위반/관리자 조치]---> SUSPENDED
ACTIVE ---[회원 탈퇴 요청]---------> CLOSED
SUSPENDED ---[조건 충족/해제]-------> ACTIVE
SUSPENDED ---[영구 해지]-----------> CLOSED
```

### 3.2 지갑 상태 (WalletStatus)

| 코드값 | 한국어 | 사용자 표시 | 정의 | 전이 규칙 |
|-------|-------|-----------|------|----------|
| `ACTIVE` | 활성 | "사용 중" (별도 표시 불필요) | 충전, 결제, 환전이 모두 가능한 정상 상태 | 초기 상태 |
| `FROZEN` | 동결 | "지갑이 동결되었습니다. 고객센터에 문의해주세요." | **외부 요인**(규제 기관 요청, 의심 거래 탐지)으로 시스템이 잠근 상태. 사용자가 직접 해제할 수 없다. 모든 출금(결제, 환전, 출금)이 차단되며 입금(충전)만 가능 | ACTIVE에서 전이. 관리자만 해제 가능 |
| `SUSPENDED` | 이용 정지 | "지갑 이용이 정지되었습니다." | **내부 요인**(정책 위반, 사용자 자발적 정지 요청)으로 이용이 제한된 상태. 조건 충족 시 해제 가능. 모든 거래(충전 포함)가 차단 | ACTIVE에서 전이. 조건 충족 시 ACTIVE로 복귀 |
| `CLOSED` | 해지 | "지갑이 해지되었습니다." | 지갑이 영구 비활성화된 상태. 잔액이 0이어야 해지 가능 | 최종 상태. 전이 불가 |

```
ACTIVE ---[의심 거래/규제 요청]-------> FROZEN
ACTIVE ---[정책 위반/사용자 요청]-----> SUSPENDED
ACTIVE ---[잔액 0 + 해지 요청]-------> CLOSED
FROZEN ---[관리자 해제]--------------> ACTIVE
SUSPENDED ---[조건 충족/해제]---------> ACTIVE
SUSPENDED ---[잔액 0 + 영구 해지]----> CLOSED
```

**FROZEN vs SUSPENDED 구분 기준**:

| 구분 | FROZEN (동결) | SUSPENDED (이용 정지) |
|------|-------------|---------------------|
| 원인 | 외부 요인 (규제, 의심 거래) | 내부 요인 (정책 위반, 사용자 요청) |
| 해제 주체 | 관리자만 해제 가능 | 조건 충족 시 자동 또는 사용자 요청 해제 |
| 충전 가능 여부 | 가능 (입금은 허용) | 불가 (모든 거래 차단) |
| 사용자 메시지 톤 | "고객센터에 문의해주세요" (수동적) | "이용이 정지되었습니다" (능동적) |

### 3.3 결제 상태 (PaymentStatus)

| 코드값 | 한국어 | 사용자 표시 | 정의 | 전이 규칙 |
|-------|-------|-----------|------|----------|
| `PENDING` | 대기 | "처리 중" (스피너 표시) | 결제 요청이 접수되어 Saga가 시작되기 전 또는 진행 중인 상태 | 초기 상태 |
| `APPROVED` | 승인 | "승인 완료" | 결제가 승인되었으나 최종 정산이 완료되지 않은 상태 | PENDING에서 전이 |
| `COMPLETED` | 완료 | "완료" (별도 배지 불필요) | 결제가 정상적으로 완료된 최종 상태 | APPROVED에서 전이 |
| `FAILED` | 실패 | "실패" (빨간색) | 결제 처리 중 오류가 발생한 상태. Saga 보상이 트리거되어 차감된 잔액이 복원된다 | PENDING 또는 APPROVED에서 전이 |
| `CANCELLED` | 취소 | "취소" (회색, 취소선) | 사용자 또는 시스템에 의해 결제가 취소된 상태 | PENDING에서만 전이 가능 |

```
PENDING ---[Saga 승인 완료]---> APPROVED ---[정산 완료]---> COMPLETED
PENDING ---[처리 오류]--------> FAILED (+ 보상 트랜잭션 실행)
PENDING ---[사용자 취소]------> CANCELLED
APPROVED --[정산 오류]--------> FAILED (+ 보상 트랜잭션 실행)
```

### 3.4 환전 상태 (ExchangeStatus)

| 코드값 | 한국어 | 사용자 표시 | 정의 | 전이 규칙 |
|-------|-------|-----------|------|----------|
| `PENDING` | 대기 | "환전 처리 중" | 환전 요청이 접수되어 처리 중인 상태 | 초기 상태 |
| `COMPLETED` | 완료 | "환전 완료" | 환전이 정상적으로 완료된 상태. EXCHANGE_OUT/EXCHANGE_IN 원장 항목이 모두 기록됨 | PENDING에서 전이 |
| `FAILED` | 실패 | "환전 실패" | 환전 처리 중 오류가 발생한 상태. 차감된 잔액이 복원된다 | PENDING에서 전이 |

> Sprint 2에서는 환전을 즉시 처리(요청 -> 완료)로 구현한다. 상태 관리는 추후 확장 시 도입한다.

### 3.5 원장 항목 유형 (EntryType)

| 코드값 | 한국어 | 사용자 표시 | 방향 | 정의 |
|-------|-------|-----------|------|------|
| `CHARGE` | 충전 | "충전" (녹색) | CREDIT (+) | 외부에서 지갑으로 KRW를 넣는 행위 |
| `WITHDRAW` | 출금 | "출금" | DEBIT (-) | 지갑에서 외부로 자금을 빼는 행위 |
| `PAYMENT` | 결제 | "결제" (빨간색) | DEBIT (-) | 가맹점에 대금을 지불하여 잔액이 차감되는 행위 |
| `PAYMENT_CANCEL` | 결제 취소 | "결제 취소" | CREDIT (+) | 결제가 취소되어 잔액이 복원되는 행위 (보상 트랜잭션) |
| `REFUND` | 환불 | "환불" | CREDIT (+) | 완료된 결제에 대해 가맹점이 대금을 돌려주는 행위 |
| `EXCHANGE_OUT` | 환전 출금 | "환전" (파란색, 통합 표시) | DEBIT (-) | 환전 시 출금 통화에서 차감되는 항목 |
| `EXCHANGE_IN` | 환전 입금 | "환전" (파란색, 통합 표시) | CREDIT (+) | 환전 시 입금 통화에 증가하는 항목 |

> **FE 표시 규칙**: EXCHANGE_OUT과 EXCHANGE_IN은 BE에서 별도 원장 항목이지만, FE 거래 내역에서는 "환전"으로 통합 표시하고 `-₩10,000 -> +$7.40` 형태로 양쪽 금액을 동시에 보여준다. 상세 보기에서 개별 항목을 확인할 수 있다.

## 4. 금액 및 통화 표현 규칙

### 4.1 지원 통화 (Currency)

| 코드 | 한국어 | 기호 | 소수점 자릿수 (Scale) | 표시 예시 |
|------|-------|------|---------------------|---------|
| `KRW` | 대한민국 원 | ₩ | 0 | ₩50,000 |
| `USD` | 미국 달러 | $ | 2 | $100.00 |
| `JPY` | 일본 엔 | ¥ | 0 | ¥1,000 |

### 4.2 금액 표시 규칙

| 상황 | 표시 형식 | 색상 | 예시 |
|------|---------|------|------|
| 입금 (CREDIT) | `+` + 통화기호 + 금액 | 녹색 (`text-green-600`) | +₩50,000 |
| 출금 (DEBIT) | `-` + 통화기호 + 금액 | 빨간색 (`text-red-600`) | -$7.99 |
| 환전 | 출금 + ` -> ` + 입금 | 파란색 (`text-blue-600`) | -₩10,000 -> +$7.40 |
| 환율 | 출금통화 1 = 입금통화 X | - | KRW 1 = USD 0.0007 |

### 4.3 금액 계산 규칙

- 모든 금액은 `BigDecimal`로 처리한다. `double`/`float` 사용 금지.
- 반올림: `RoundingMode.HALF_UP`
- 저장: `DECIMAL(18,4)` (내부 정밀도)
- 표시: 통화별 scale에 따라 포맷 (KRW/JPY: 정수, USD: 소수점 2자리)

## 6. 혼용 금지 용어

아래 용어 쌍에서 LemonPay 프로젝트는 **왼쪽 용어만 사용**한다. 오른쪽 용어는 코드, 문서, 커뮤니케이션 어디에서도 사용하지 않는다.

| 사용 용어 | 사용 금지 | 사유 |
|----------|---------|------|
| **충전** (Charge) | 입금, 탑업, 리차지 | 선불 지갑 서비스 업계 표준 (카카오페이, 네이버페이 등). "입금"은 은행 계좌 느낌 |
| **결제** (Payment) | 지불, 구매 | "결제"가 핀테크 서비스의 표준 용어. "지불"은 법률 용어, "구매"는 상품 구매에 한정 |
| **환전** (Exchange) | 환율 변환, 통화 변환 | "환전"이 금융 업계 표준이며 사용자에게 직관적 |
| **가맹점** (Merchant) | 상점, 판매자 | BE 코드에서는 `merchant`를 사용. 단, UI에서는 상호명만 직접 표시하고 "가맹점"이라는 단어를 노출하지 않음 |
| **결제처** | 가맹점 (UI 한정) | UI에서 레이블이 필요한 경우 "결제처" 사용. "가맹점"은 B2B 느낌, "상점"은 오프라인 느낌 |
| **원장** (Ledger) | 장부, 거래 기록부 | DDD 및 핀테크 업계 표준 용어 |
| **대사** (Reconciliation) | 정합성 검증, 잔액 검증 | 금융 업계 표준 용어. "정합성 검증"은 너무 일반적 |
| **동결** (FROZEN) | 잠금, 차단, 블록 | 외부 요인(규제, 의심 거래)에 의한 비활성화. "잠금"은 동시성 락과 혼동 |
| **이용 정지** (SUSPENDED) | 일시 중지, 비활성화 | 내부 요인(정책 위반, 사용자 요청)에 의한 비활성화 |
| **해지** (CLOSED) | 삭제, 탈퇴, 폐쇄 | 금융권 관례. "삭제"는 물리 삭제 오해 유발 (논리 삭제만 사용). 단, 사용자 표시에서는 "탈퇴"도 허용 |
| **보상** (Compensation) | 롤백, 되돌리기 | Saga 패턴의 공식 용어. "롤백"은 DB 트랜잭션 롤백과 혼동 |
| **스냅샷 잔액** | 캐시 잔액, 현재 잔액 | `wallet_balance.balance`를 지칭하는 정확한 용어. "캐시"는 Redis 등 인프라 캐시와 혼동 |
| **멱등성 키** (Idempotency Key) | 중복 방지 키, 고유 키 | API 설계 표준 용어 |
| **`BASC`** (상품 코드) | BASIC, BSC, BASE | ISO 20022 스타일 4자리 코드 사용. `BASIC` 등 풀네임이나 다른 약어 사용 금지 |
| **`TRVL`** (상품 코드) | TRAVEL, TRV | 동일 사유. 4자리 ISO 스타일 통일 |
| **`PREM`** (상품 코드) | PREMIUM, PRM | 동일 사유. 4자리 ISO 스타일 통일 |
